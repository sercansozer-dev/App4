GoPxL SDK Documentation

Comprehensive guide for the GoPxL C++ SDK and GoPxL .NET SDK – open-source libraries for programmatically accessing and controlling Gocator devices.

Quick Start

  1. Extract the SDK archive
  2. Navigate to GoPxL_SDK_Cpp/Gocator/GoPxLSdk (C++) or GoPxL_SDK_dotNet/Gocator/GoPxLSdkNet (.NET)
  3. Visual Studio solution (.sln) files are included for Windows. For Linux (C++ only), run python -m generate to create makefiles
  4. Make any necessary code changes (e.g., update sensor IP if different from default)
  5. Build using Visual Studio (Windows) or Make (Linux) for C++. For .NET, open GoPxLSdkNet.sln in Visual Studio (Windows only)
  6. Run samples from GoPxL_SDK_Cpp/bin (C++ Windows/Linux) or GoPxL_SDK_dotNet/bin (.NET Windows), or run directly from Visual Studio

Overview

The GoPxL Software Development Kit (GoPxL SDK) is available in two variants:

C++ SDK

An open-source C++ library for programmatically accessing and controlling Gocator devices. Provides a modern C++ interface to the GoPxL REST API.

Platform Support: Windows, Linux (x64, ARM)

Architecture

The C++ SDK implements a REST-like wrapper over a custom binary protocol, built on top of the kApi (K Framework) and GoApi (Gocator API) foundations. The architecture features a dual-threaded I/O model with separate read/write threads to prevent blocking on network operations. Communication uses MessagePack and JSON serialization over TCP with binary framing via kApi's kSerializer.

The SDK employs nlohmann/json for JSON handling (forward-declared in headers for ABI stability), std::shared_ptr for exception-safe resource management, and a Promise/Future pattern via std::shared_future for synchronous API layering over asynchronous transport. Request-response correlation is managed through auto-incrementing 64-bit IDs with transaction queuing via std::deque.

Data streaming leverages the GoPxL Data Protocol (GDP) on a separate TCP channel (default port 3601) for high-bandwidth binary data transfer. The protocol supports 15+ message types (profiles, surfaces, measurements, features, etc.) with scale factors converting millimeters to nanometers. Thread synchronization uses std::mutex with RAII lock guards, and kSemaphore for non-blocking queue signaling.

The SDK is cross-platform (Windows x64, Linux x64/ARM7/ARM64) and builds as a shared library with configurable timeouts per operation class, supporting both blocking and async I/O patterns.

.NET SDK

A managed C# library for programmatically accessing and controlling Gocator devices. Provides a .NET interface to the GoPxL REST API.

Platform Support: Windows only (.NET Framework 4.7.2+)

Architecture

The .NET SDK targets .NET Framework 4.8 with C# 7.3. It uses MessagePack (3.1.3) and Newtonsoft.Json (13.0.3) for dual-protocol serialization. The architecture implements a layered REST wrapper with binary carrier protocol, featuring custom JObjectFormatter for seamless integration between MessagePack and dynamic JSON objects (JObject).

Thread safety is ensured through explicit locking via object _commandMutex and BlockingCollection for producer-consumer patterns. The SDK uses a dual-threaded I/O model with dedicated read/write threads, and employs TaskCompletionSource for the Promise/Future pattern which enables synchronous API calls over asynchronous transport.

Binary I/O is handled through custom GoBinaryReader/GoBinaryWriter with length-prefixed framing and nested section support via stack-based tracking. System.Memory (4.6.3) provides Span<T> primitives for high-performance data handling. The GDP stream handler supports both synchronous polling (Client.Poll()) and asynchronous callback-driven modes with background thread pools.

Discovery uses UDP broadcast (port 3320) with magic signature validation (0x4C58504F47494D4C). Request-response correlation utilizes transaction queues with per-operation timeouts (default 3-15 seconds). The protocol implements retry logic, graceful degradation, and comprehensive exception handling via GoChannelException and GoRequestException.

Platform support is Windows-only (x64/AnyCPU), with dependency on Microsoft.Bcl.AsyncInterfaces for modern async patterns in .NET 4.8.

Common Features

Both SDKs enable developers to:

  • Discover sensors on the network
  • Configure measurement tools
  • Control sensor operations (start/stop scanning, alignment, etc.)
  • Receive real-time scan data (profiles, surfaces, images, stamp, etc.)
  • Manage sensor jobs and configurations

Building the SDK

C++ Build Instructions

The SDK uses a Python-based build system generator that creates platform-specific build files (Visual Studio solutions for Windows, GNU Makefiles for Linux).

Windows

  1. Navigate to the SDK directory:

    cd GoPxL_SDK_Cpp\Gocator\GoPxLSdk

  2. Generate Visual Studio solution:

    python -m generate

    This creates GoPxLSdk.sln in the parent directory (GoPxL_SDK_Cpp/Gocator/).

  3. Open and build in Visual Studio:

    cd ..
start GoPxLSdk.sln
    • Select Debug or Release configuration
    • Choose x64 platform
    • Build the entire solution (Build → Build Solution or Ctrl+Shift+B)

  4. Build output locations:

    • Executables: GoPxL_SDK_Cpp/bin/win64/ (Release) or GoPxL_SDK_Cpp/bin/win64d/ (Debug)
    • Libraries: GoPxL_SDK_Cpp/lib/win64/ (Release) or GoPxL_SDK_Cpp/lib/win64d/ (Debug)

Alternative - Command Line Build:

cd GoPxL_SDK_Cpp\Gocator
msbuild GoPxLSdk.sln /p:Configuration=Release /p:Platform=x64

Linux

  1. Navigate to the SDK directory:

    cd GoPxL_SDK_Cpp/Gocator/GoPxLSdk

  2. Generate Makefiles:

    python -m generate

    This creates GoPxLSdk-Linux_X64.mk in the parent directory.

  3. Build the SDK and samples:

    cd ..
make -f GoPxLSdk-Linux_X64.mk

    For parallel builds (faster):

    make -j$(nproc) -f GoPxLSdk-Linux_X64.mk

  4. Build output locations:

    • Executables: GoPxL_SDK_Cpp/bin/linux_x64/
    • Libraries: GoPxL_SDK_Cpp/lib/linux_x64/libGoPxLSdk.so

Clean build:

make -f GoPxLSdk-Linux_X64.mk clean

.NET Build Instructions

The SDK uses a Python-based build system generator that creates Visual Studio solution and project files.

  1. Navigate to the SDK directory:

    cd GoPxL_SDK_dotNet\Gocator\GoPxLSdkNet

    python -m generate

    This creates GoPxLSdkNet.sln in the parent directory (GoPxL_SDK_dotNet/Gocator/).

  3. Open in Visual Studio:

    cd ..
start GoPxLSdkNet.sln

  4. Select build configuration:

    • Configuration: Debug or Release
    • Platform: Any CPU or x64

  5. Build the solution:

    • Menu: Build → Build Solution (or Ctrl+Shift+B)
    • All projects (SDK library + 17 samples) will be built

  6. Build output locations:

    • Debug builds: GoPxL_SDK_dotNet/bin/win64d/
    • Release builds: GoPxL_SDK_dotNet/bin/win64/
    • Assemblies: GoPxLSdkNet.dll and sample executables
    • Dependencies: Required DLLs are copied automatically

Alternative - Command Line Build:

cd GoPxL_SDK_dotNet\Gocator
msbuild GoPxLSdkNet.sln /p:Configuration=Release /p:Platform="Any CPU"

Clean Build:

  • Visual Studio: Build → Clean Solution
  • Command line: msbuild GoPxLSdkNet.sln /t:Clean

Note: The python -m generate step is only required when building from source. SDK packages distributed from LMI Technologies include pre-generated project files.

Using the REST API

Main Classes and Methods

The GoPxL SDK provides several key classes for interacting with sensors via the REST API:

  • GoSystem - Primary class for managing sensor connections and system state. Provides high-level methods like Connect(), Start(), Stop(), and Disconnect(). Access the REST client via system.Client()
  • GoRestClient - Handles all REST API communication. Accessed through system.Client(), it provides methods for:
    • Read(path) - Retrieve resource content (for Readable resources)
    • Update(path, payload) - Modify resource properties (for Observable resources)
    • Create(path, payload) - Create new collection items (for Collection resources)
    • Delete(path) - Remove resources (for Collection resources)
    • Call(path) - Invoke actions and commands (for Callable resources)
    • Subscribe(path) / Unsubscribe(path) - Manage event notifications (for Observable resources)
  • GoJson - JSON manipulation class for constructing payloads and parsing responses. Use At(path) to navigate nested properties and Get<T>() to extract typed values
  • GoGdpClient - Handles high-bandwidth binary data streaming over the GDP protocol (port 3601). Use Connect() and ReceiveDataSync() to receive profiles, surfaces, and measurements

How REST API is Used Under the Hood

The SDK abstracts REST API interactions through high-level methods, but understanding the underlying REST operations is crucial for advanced scenarios. Every SDK operation maps to REST API calls:

  • Resource paths - The SDK uses hierarchical URI paths like /scan/scanners/scanner-0/sensors/sensor-0 to address specific resources (scanners (sensor groups), sensors, tools, control protocols). Each path follows the format /collection/item-id/subcollection/subitem-id
  • Resource types and operations - Resources in the REST API are categorized by type, which determines the available operations:
    • Standard - Resources that contain JSON properties and can be read using Read() and modified using Update()
    • Observable - Resources that can be monitored via Subscribe() for asynchronous change notifications. Unless otherwise stated, all resources are Observable
    • Collection - Resources that contain multiple items, supporting Create() to add new items and Delete() to remove items
    • Callable - Action/command resources invoked using Call() (e.g., /system/commands/start, /scan/scanners/scanner-0/actions/trigger)
  • Response handling - All REST operations return responses containing status (1=Success), payload (JSON data), and path. The SDK's CheckResponse() method validates the status and throws exceptions for errors, while Payload() extracts JSON data for parsing: 
    GoJson response = system.Client().Read("/system").GetResponse().Payload();
  int runState = response.At("runState").Get<int>();
  • Subscriptions and notifications - For event-driven applications, use system.Client().Subscribe(path) to receive asynchronous notifications when Observable resources change (e.g., sub /system notifies when runState transitions, sub /tools/* notifies when any tool updates). This eliminates polling overhead
  • GDP integration - While configuration uses REST API over the control port, high-bandwidth data streaming uses the separate GDP protocol (port 3601). Enable GDP via REST (system.Client().Update("/controls/gocator", {"enabled":true})), then use GoGdpClient to receive binary messages (profiles, surfaces, measurements, images) with message types identified by enum values rather than parsing JSON
  • Error handling - REST errors return structured JSON in payload.errors with status codes and descriptions. The SDK throws GoRequestError for failed REST commands (invalid paths, timeout) and GoChannelError for connection issues, allowing targeted exception handling

For developers extending SDK functionality, directly using system.Client().Read(), Update(), Call(), etc. with custom resource paths enables accessing any REST API feature not wrapped by high-level SDK classes. Consult the REST API reference documentation (located in the "doc" folder) for complete information including:

  • Resource Reference - Complete list of all available resources organized by category:
    • General resources (system, version, jobs, replay)
    • Scanner and sensor configuration (scan engines for G2/G3/G4/G5 sensors)
    • Tool resources (measurement tools, inputs/outputs)
    • Control protocols (GDP, HMI, Modbus, EtherNet/IP, PROFINET, Digital Output, Ethernet ASCII)
    • Metrics resources (performance statistics and system health)
    • Routine resources (alignment, calibration workflows)
  • User Guide - Detailed explanations of REST API concepts:
    • Resource design and request methods (Create, Read, Update, Delete, Call, Subscribe)
    • Resource types and traits (Standard, Observable, Collection, Callable)
    • Connection and message framing (TCP protocol, MessagePack/JSON serialization)
    • Notification and streaming (subscription patterns, event handling)
    • Schema and data types (property validation, units, enums)
    • GDP (GoPxL Data Protocol) for high-bandwidth binary data streaming
    • HAL (Hypertext Application Language) for resource discovery
    • Versioning and compatibility guidelines

SDK Workflow Overview

A typical GoPxL SDK application follows this workflow pattern, as demonstrated in the ConfigureSensor sample:

  1. Initialize the SDK - Construct the GoApiLib framework to initialize the underlying kApi and GoApi foundations that handle binary protocol communication: 
    kAssembly goApiLib = kNULL;
  GoApiLib_Construct(&goApiLib);
  2. Connect to the sensor - Create a GoSystem object, set the sensor IP address and control port (default 3600), then call system.Connect() to establish a TCP connection: 
    GoSystem system;
  kIpAddress systemIpAddress;
  kIpAddress_Parse(&systemIpAddress, "192.168.1.10");
  system.SetAddress(systemIpAddress);
  system.SetControlPort(3600);
  system.Connect();
  3. Configure the sensor - Update scanner (aka sensor group) and sensor settings through REST API calls:
    • Read current configuration: system.Client().Read("/scan/scanners/scanner-0")
    • Update parameters: system.Client().Update(path, payload) where payload is JSON containing the properties to modify (e.g., trigger source, exposure values, active area dimensions)
    • The SDK uses patch semantics by default, meaning you only specify properties you want to change—the rest remain unchanged
    GoJson payload = GoJson(R"({"parameters": {"exposureSettings": {"singleExposure": 1200}}})");
  system.Client().Update("/scan/scanners/scanner-0/sensors/sensor-0", payload).CheckResponse(20000);
  4. Configure tools and outputs (if needed) - Create measurement tools via system.Client().Create("/tools", {"type": "ProfileArea"}), configure tool parameters, and map outputs to control protocols: 
    GoJson createPayload = GoJson(R"({"type": "ProfileArea"})");
  GoJson response = system.Client().Create("/tools", createPayload).GetResponse().Payload();
  string toolId = response.At("id").ToString();
  5. Start data acquisition - Call system.Start() which internally invokes the REST API command call /system/commands/start to transition the system to Running state: 
    system.Start();
  6. Receive data - For measurement/profile/surface data, connect a GoGdpClient to the GDP port (default 3601) and call ReceiveDataSync() to retrieve binary data streams: 
    system.Client().Update("/controls/gocator", GoJson("{\"enabled\":true}")).CheckResponse(20000);
  auto gdpClient = std::make_unique<GoGdpClient>();
  gdpClient->Connect(system.Address(), system.GdpPort());
  gdpClient->ReceiveDataSync(20000);

  const GoDataSet& dataSet = gdpClient->DataSet();
  for (size_t i = 0; i < dataSet.Count(); i++) {
      const auto& msg = dataSet.GdpMsgAt(i);
      if (msg.Type() == MessageType::MEASUREMENT) {
          const auto& measurementMsg = static_cast<const GoGdpMeasurement&>(msg);
          std::cout << "Value: " << measurementMsg.Value() << std::endl;
      }
  }
  7. Stop and cleanup - Call system.Stop() and system.Disconnect() to halt acquisition and close connections: 
    system.Stop();
  system.Disconnect();
  kDestroyRef(&goApiLib);

Running the Samples

Configuration

Before running any sample, you must update the sensor connection parameters in the source code:

C++ Example (from any sample .cpp file):

const std::string SENSOR_IP = "192.168.1.10";  // Update with your sensor's IP address
const uint16_t CONTROL_PORT = 3600;             // Default control port is 3600

.NET Example (from any sample .cs file):

private const string SENSOR_IP = "192.168.1.10";  // Update with your sensor's IP address
private const ushort CONTROL_PORT = 3600;          // Default control port is 3600

To find your sensor's IP address:

  1. Use the Discover sample to scan the network
  2. Use GoPxL Manager application
  3. Check the sensor's display or web interface
  4. Check your DHCP server for assigned addresses

Important: After changing these values, you must rebuild the sample.

Troubleshooting

Build Issues

Problem: "Python not found" or "python command not recognized"

  • Windows: Install Python from https://python.org and ensure it's in your PATH
  • Linux: sudo apt install python3 python3-pip

Problem: "Cannot find Visual Studio" (C++)

  • Ensure Visual Studio 2017 or later is installed with C++ workload
  • Run from "Developer Command Prompt for VS"

Problem: "g++ not found" (Linux)

sudo apt update
sudo apt install build-essential

Problem: ".NET Framework 4.7.2 not found"

  • Install .NET Framework 4.7.2 Developer Pack
  • Or retarget the solution to your installed version

Problem: "Permission denied" when running Linux samples

chmod +x GoPxL_SDK_Cpp/bin/linux_x64/*

Runtime Issues

Problem: "Connection refused" or "Cannot connect to sensor"

  • Verify sensor IP address is correct
  • Check sensor is powered on and network cable is connected
  • Ping the sensor: ping 192.168.1.10
  • Check firewall settings are not blocking ports 3600-3601
  • Ensure sensor is in remote mode (accelerated or buddied)

Problem: "Timeout waiting for data"

  • Verify sensor is in scanning mode
  • Check sensor is properly configured with a valid job
  • For samples requiring live data, ensure sensor can trigger and scan

Problem: Linux samples crash with "error while loading shared libraries"

export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:$PWD/../../lib/linux_x64

Or add permanently to ~/.bashrc:

echo 'export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:/path/to/GoPxL_SDK_Cpp/lib/linux_x64' >> ~/.bashrc
source ~/.bashrc

Problem: "No sensors found" when running Discover

  • Ensure sensor and PC are on the same network/subnet
  • Check UDP broadcast is not blocked by firewall
  • Try direct connection with IP address instead

Problem: Windows Firewall blocks the application (.NET)

  • Click "Allow access" when prompted
  • Or manually add exception: Windows Defender Firewall → Allow an app

Common Configuration Errors

Problem: Sensor rejects connection

  • Another application may have exclusive control
  • Close GoPxL Manager or other SDK applications
  • Reboot the sensor if necessary

Problem: Data reception fails

  • Verify sensor is configured with valid scan settings
  • Check sensor firmware version compatibility
  • Ensure sufficient network bandwidth
